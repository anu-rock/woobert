## What this is

**Hoobert** is a merchant-facing agentic command bar for WooCommerce, packaged as a
production WordPress plugin. A merchant presses `⌘K`/`Ctrl-K` in wp-admin, types a
natural-language request, and Hoobert maps it to a WooCommerce **REST API v3** (`wc/v3`)
tool call that the plugin executes server-side under the admin's own capabilities. The
natural-language step is handled by a fine-tuned function-calling model (Fern) reached
through an inference endpoint. See [`README.md`](README.md) for the request flow.

Hoobert is intended to ship via the WordPress plugin directory, so treat it as a real,
shippable product: no "demo" framing in code, UI, or docs.

## Inference contract

- The plugin POSTs to a single inference endpoint (the full URL, e.g.
  `https://fernfly.com/api/p/27/infer`), set on the settings page (WooCommerce -> Hoobert).
- Request: `POST {endpoint}` with header `X-Api-Key: {api_key}` and JSON body
  `{ utterance, meta }`. The `utterance` is the merchant query with page context folded
  in; `meta` carries the raw context.
- Response: `{ calls: [{ name, arguments }], ... }`. Out-of-scope requests come back as
  `{ calls: [], out_of_scope: true, reply }`. The endpoint owns the registered tool set,
  so the plugin does **not** send tools or a model id.
- The client lives in `includes/class-fern-client.php` (`Hoobert_Fern_Client`).

## Architecture decisions (and why)

- **REST API v3, not the Store API.** The journeys (refunds, create product, order notes,
  top customers, coupons) are merchant/admin operations that live in `wc/v3`. The
  customer-facing Store API (`wc/store/v1`) can't do them. The model is trained on REST v3
  tool schemas.
- **WordPress plugin, executing REST server-side.** Hoobert is a WP plugin, not a
  WooCommerce React (SPA) extension. This keeps the training target the stable REST API
  rather than internal `@wordpress/data` store actions, and lets the PHP proxy hold the
  inference key and run tool calls in-process (`rest_do_request`) under the admin's own
  capabilities, so no API keys in the browser. Consumer key/secret pairs are still
  generated by setup for external/curl testing.
- **Resolve then execute, with confirmation.** `/hoobert/v1/resolve` returns the chosen
  tool call without running it; the UI previews it and, for tools flagged `x-woo.confirm`,
  requires a confirm click before `/hoobert/v1/execute` runs it.

## Project layout

- `plugin/hoobert/` — `src/` (command-palette front-end, built with `@wordpress/scripts`), `includes/`
  (PHP: `class-rest-proxy`, `class-executor`, `class-fern-client`, `class-tools`,
  `class-settings`). Main file: `hoobert.php`. PHP classes are prefixed `Hoobert_`;
  plugin constants `HOOBERT_*` (VERSION/PATH/URL); the inference endpoint URL + API key
  are stored as plugin options (set under WooCommerce -> Hoobert). REST namespace
  `hoobert/v1`; text domain `hoobert`.
- `plugin/hoobert/tools.json` — the shipped tool set. The executor reads each tool's private
  `x-woo` block to dispatch the REST call. The canonical training copy lives in the Fernfly
  project; keep the two in sync when tools change.
- `docker-compose.yml`, `scripts/setup.sh`, `scripts/seed-sample-data.php`,
  `scripts/generate-api-key.php`.
- `assets-src/` — sources for the plugin-directory artwork (owl SVG, banner HTML);
  `.wordpress-org/` — the generated icons/banners plus hand-added screenshots, copied
  to SVN `assets/` on release. Regenerate with `node scripts/build-wporg-assets.mjs`;
  never hand-edit the PNGs. The owl also ships inside the plugin at
  `plugin/hoobert/assets/hoobert-owl.svg` for the settings screen, and its Flaticon
  attribution must stay on that screen, in `readme.txt`, and in `README.md`.

## Shipping to WordPress.org

[`docs/wordpress-org-submission.md`](docs/wordpress-org-submission.md) is the playbook.
Things to keep in mind when changing anything user-facing:

- **Verify with the real tool, not by eye.** `docker compose run --rm --entrypoint wp
  wpcli plugin check hoobert --exclude-directories=node_modules,src,scripts,build`.
  It should report zero errors; the only expected warnings are the naming ones.

## Commands

```bash
# Front-end bundle (run inside plugin/hoobert)
npm install
npm run build        # wp-scripts build
npm run start        # watch mode

# Local stack (repo root)
docker compose up -d                          # up + auto-provision via the wpcli service (idempotent)
docker compose logs -f wpcli                  # watch provisioning; API keys are printed here
docker compose run --rm wpcli                 # re-run provisioning (e.g. after building the plugin)
docker compose run --rm --entrypoint wp wpcli <cmd>   # ad-hoc WP-CLI
```

Images are pinned in `docker-compose.yml` (WordPress, MariaDB, WP-CLI) so every contributor
runs the same versions; bump them deliberately alongside the plugin's "Tested up to" header.

## Conventions & preferences

**Git**
- Commit **directly to `main`** — no feature branches.
- **Never commit without the user reviewing first**, and only commit when explicitly asked. Push only when asked.
- Always make a **new commit** for follow-ups — never `git commit --amend`, even after review.
- [Conventional Commits](https://www.conventionalcommits.org/): `type: lowercase imperative description`. Types: `feat`, `fix`, `refactor`, `chore`, `ci`, `docs`. Confirm state with `git status`/`git diff` before committing.

**Code style**
- Document exported/non-trivial functions and file headers with **block doc comments** (`/** … */` in JS; `/** … */` PHPDoc in PHP), covering what it does and any non-obvious *why*. Keep them terse. Inline `//` comments are fine inside a body.
- Keep added comments to **1–2 lines**. Derive computed state (counts, flags) server-side, not in the front-end.
- PHP: WordPress coding standards, guard files with `if ( ! defined( 'ABSPATH' ) ) exit;`, escape output, check `manage_woocommerce` on admin routes.

**Writing style (docs, UI copy, messages)**
- **No em dashes.** Use a hyphen, period, or comma instead.
- Strunk & White concision: short, direct sentences; cut filler.

## Notes for future work

- The tool set is deliberately capped at ~28 tools covering core journeys; grow it deliberately (each tool widens the fine-tune surface).
- `current_order_id` / `current_product_id` context is filled from the wp-admin screen (`Hoobert_Rest_Proxy::current_page_context()`); this is what resolves "this order" / "current product". When generating training data, include that context in prompts.
- Reports tools (`get_sales_report`, `get_top_sellers`) use the legacy `wc/v3/reports/*`; `get_top_customers` uses the `wc-analytics` namespace. Analytics must be enabled (setup does this).
